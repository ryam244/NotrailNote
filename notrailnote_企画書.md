# AI生成コンテンツ＆アイデア管理ツール 事業計画書 (統合・完全版)

## 1. プロジェクト概要

本企画は、生成AIの出力結果や個人のアイデア、会議の議事録などのテキスト情報を効率的に管理し、その進化の過程を追跡・活用するためのアプリケーションを提案するものです。**まずはiOSアプリとして開発を進め、その後、Android、Web（PWA）へと展開することで、PC、スマートフォン、タブレットのあらゆるデバイスでネイティブアプリのような体験を提供することを目指します。** 開発者側のインフラコストを極限まで抑えるため、**自前サーバーを持たず、ユーザーのGitHubリポジトリに直接データを保存する**設計を基本とします。Markdown形式での柔軟な編集、バージョン管理システム（GitHub）との連携、そして音声データからのテキスト化機能を通じて、ビジネスパーソンやクリエイターの生産性向上に貢献します。

### 1.1. 目的

生成AIの普及により、テキストコンテンツの生成は容易になりましたが、その管理や再利用、バージョン管理は依然として課題です。本アプリは、AI生成テキスト、人間のアイデア、音声データからの書き起こしテキストを一元的に管理し、その「進化の過程」を可視化することで、ユーザーがより質の高いコンテンツを効率的に生み出すことを支援します。特に、**iOSアプリとして先行開発することで、iPhone/iPadユーザーに最適化された体験を提供し、GitHubをデータストレージとして活用することで、開発者側のデータ管理コストをゼロに抑えます。**

### 1.2. ターゲットユーザー

本アプリは、主に以下の二つのターゲット層を想定しています。

1. **初期ターゲット（最優先）**: **GitHubに不慣れだが、ドキュメントのバージョン管理や変更履歴の追跡に課題を感じているビジネスパーソン。** 会議の議事録、企画書、アイデアメモの作成・管理、プロンプトエンジニアリングを行う方々が、直感的な操作でAI生成コンテンツや自身のアイデアを効率的に管理できることを目指します。
2. **機能拡張ターゲット**: プロンプト管理や高度な連携を求めるエンジニアやAI活用層。拡張性の高い機能と、Gitの概念を理解した上での柔軟なカスタマイズ性を提供します。

### 1.3. 提供価値

本アプリは、以下の主要な価値をユーザーに提供します。

- **アイデアの可視化と進化**: テキストの差分履歴を追うことで、アイデアがどのように発展したかを明確に把握できます。
- **堅牢かつシンプルなコンテンツ管理**: Markdown形式とGitHubによる堅牢なバージョン管理を、Gitを意識させないシンプルなUI/UXで実現します。
- **AI活用促進**: AIの出力結果を資産として蓄積・活用し、プロンプトの改善や再利用を促します。
- **多様な情報源の統合**: 音声データを含む様々な情報源からのテキストを一元的に管理できます。

## 2. 主要機能

### 2.1. Markdownベースのコンテンツ作成・編集

- **リッチテキストエディタ**: Markdown記法に対応した直感的で使いやすいエディタを提供します。
- **リアルタイムプレビュー**: 編集中のMarkdownがどのように表示されるかをリアルタイムで確認できます。
- **段落単位の編集**: ドキュメント全体ではなく、特定の段落のみを対象とした編集・更新が可能です。

### 2.2. バージョン管理と履歴追跡

- **自動保存と履歴**: 編集内容を自動で保存し、変更履歴を自動的に記録します。
- **差分表示**: 過去のバージョンとの差分を視覚的に分かりやすく表示します（追加・削除・変更箇所をハイライト）。
- **段落単位のバージョン管理**: 特定の段落に特化したバージョン履歴を管理し、その段落のみを過去の状態に戻す機能を提供します。
- **スナップショット機能**: 特定の時点のドキュメント全体、または特定の段落の状態を「スナップショット」として保存し、後から参照・復元できるようにします。

### 2.3. 外部サービス連携

- **GitHub連携**: MarkdownファイルをGitHubリポジトリに同期し、Gitのバージョン管理機能を活用します。プルリクエストを通じた共同編集フローもサポートします。
- **音声認識サービス連携**: 既存の音声ファイル（mp3, wav, mp4など）をアップロードすると、AI音声認識サービス（例: Google Cloud Speech-to-Text, OpenAI Whisper）を通じてテキストに変換し、Markdownとして保存します。
- **クラウドストレージ連携**: Google Drive, Dropbox, Notion, Obsidianなどの個人知識管理ツールやクラウドストレージサービスとの連携を検討し、ユーザーの既存ワークフローへの統合を容易にします。

### 2.4. プロンプト管理機能

- **プロンプトテンプレート保存**: よく使うAIプロンプトをテンプレートとして保存・管理し、再利用を容易にします。
- **プロンプトと出力の紐付け**: AIの出力結果に対し、使用したプロンプト、モデル名、パラメータなどのメタデータを自動的に付与し、紐付けて管理します。
- **プロンプトのバージョン管理**: プロンプト自体もバージョン管理の対象とし、プロンプトの改善履歴を追跡できるようにします。

### 2.5. その他の機能

- **検索機能**: ドキュメント内のキーワード検索に加え、メタデータや履歴情報も対象とした高度な検索機能を提供します。
- **タグ・カテゴリ分類**: ドキュメントを整理するためのタグ付けやカテゴリ分類機能。
- **クロスプラットフォーム対応**: **まずはiOSアプリとして開発し、その後Androidアプリ、Webアプリケーション（PWA）へと順次展開します。**

## 3. 競合分析と差別化戦略

本アプリは既存のツールと競合する部分もありますが、独自のポジションを確立することで市場に参入します。

| 競合サービス | 強み | 弱み・本アプリの勝機 |
| --- | --- | --- |
| **Notion, Obsidian** | 高機能なナレッジベース、豊富なテンプレート、柔軟なデータ構造。コミュニティによるプラグインエコシステムが強力。 | **バージョン管理がGitネイティブではない。** GitHubとの連携は可能だが、プラグインの導入や設定が複雑で、非エンジニアにはハードルが高い。 本アプリは、Gitによる堅牢なバージョン管理を、専門知識不要で直感的に利用できる点で優位性を持つ。 |
| **GitHub Projects, Issues** | 開発プロジェクト管理に特化。IssueとPull Requestの連携が強力で、開発フロー全体を管理可能。 | **ドキュメント作成・管理ツールではない。** Markdownは利用できるが、アイデアのブレインストーミングや長文コンテンツの執筆・編集には不向き。 本アプリは、コンテンツそのものの生成・編集・進化に焦点を当てる。 |
| **一般的なMarkdownエディタ** | 執筆に特化しており、軽量で高速なものが多い。 | バージョン管理機能が弱い、または存在しない。GitHub連携も手動での操作が必要な場合が多い。 |

**差別化戦略:**

本アプリは、**「ビジネスパーソン向けの優れたUX」**と**「Gitネイティブな堅牢なバージョン管理」**の交差点にポジションを確立します。Notionのような使いやすさと、GitHubの強力なバージョン管理能力を、専門知識なしで誰もが享受できる世界を目指します。

## 4. フェーズ別開発工程ロードマップ

本アプリケーションの開発は、MVP（Minimum Viable Product）を早期にリリースし、ユーザーからのフィードバックを基に段階的に機能を拡張していくアジャイルなアプローチを採用します。これにより、市場のニーズに迅速に対応し、開発リスクを最小限に抑えます。

### フェーズ1: iOS MVP (Minimum Viable Product) - 4ヶ月

**目的**: iOSアプリとしてコア機能を提供し、基本的なユーザー体験とGitHub連携の検証を行う。

**主要機能**:

- **ユーザー認証**: GitHub OAuthによるログイン。
- **Markdownエディタ**: 基本的なMarkdown記法に対応したネイティブUIのエディタ。リアルタイムプレビュー機能。
- **ファイル管理**: アプリ内でのMarkdownファイルの作成、保存、削除。
- **GitHub連携（基本）**: アプリ内のMarkdownファイルを指定したGitHubリポジトリにプッシュ（新規作成・更新）。ユーザーはリポジトリとブランチを選択するのみで、Gitコマンドは意識させない。
- **差分表示**: GitHubのコミット履歴を利用した簡易的な差分表示（ファイル全体）。
- **無料プランの提供**: ファイル数、文字数制限付き。

**技術スタック**: React Native (iOS), GitHub API (直接連携), Firebase Authentication (認証)。バックエンドは最小限のAPIプロキシ（AWS Lambda/Google Cloud Functions）を検討。

### フェーズ2: 機能拡張とAndroid/Web展開 - 6ヶ月

**目的**: ユーザーフィードバックを反映し、利便性と独自性を高める機能を追加し、AndroidアプリおよびWeb（PWA）版を展開する。

**主要機能**:

- **段落単位の編集・差分表示**: ユーザーの要望に応じた段落ごとの編集と、その差分を視覚的に分かりやすく表示する機能。
- **音声認識連携**: 既存の音声ファイル（mp3, wav）をアップロードし、AI音声認識サービス（OpenAI Whisper APIなど）を通じてテキスト化し、Markdownとして保存する機能。**音声ファイルはテキスト化後、即時破棄し保存しない。**
- **プロンプト管理機能**: よく使うプロンプトのテンプレート保存、AI出力との紐付け（メタデータ自動付与）。
- **有料プランの導入**: Basic, Proプランの提供。
- **UI/UX改善**: GitHub連携時の操作フローの簡略化、エラーメッセージの改善。
- **Androidアプリ版のリリース**: React Nativeのコードベースを活用し、Androidアプリをリリース。
- **Web（PWA）版のリリース**: React Native Webを活用し、PWAとしてWeb版をリリース。

**技術スタック**: フェーズ1に加えて、OpenAI Whisper API (低コストモデル), LLM API (GPT-4o-miniなど低コストモデルを厳選)。API呼び出しは、最小限のAPIプロキシ（AWS Lambda/Google Cloud Functions）経由で、制作側がコストを管理。

### フェーズ3: 高度な連携とチーム機能強化 - 9ヶ月

**目的**: チームでの利用や多様なデバイスからのアクセスに対応し、市場競争力を強化する。

**主要機能**:

- **GitHub連携（高度）**: プルリクエストの作成・レビュー機能、特定ブランチへの自動マージ設定。
- **チーム機能**: 複数ユーザーでのリポジトリ共有、ロールベースのアクセス制御。
- **外部サービス連携**: Notion, Obsidian, Google Driveなど、他のクラウドサービスとの同期機能。
- **Businessプランの提供**: チーム機能、優先サポート。

**技術スタック**: フェーズ2に加えて、各種クラウドAPI。

## 6. 収益モデルとコスト試算、ユーザー予測 (サーバーレス構成版)

### 6.1. 前提条件と仮定

- **ターゲットユーザー**: 初期はGitHubに不慣れなビジネスパーソン、機能拡張でエンジニア/AI活用層。
- **音声データ処理**: 既存ファイルのアップロード。**音声ファイルはテキスト化後、即時破棄し保存しない。**
- **LLM利用**: 音声・画像データのテキスト化とその初期整形にのみ利用。GPT-4o-miniなどの低コストモデルを厳選。

### 6.2. 収益モデル

- **無料プラン**: ファイル数、文字数制限あり。バナー広告表示。
- **有料プラン**: 月額サブスクリプション制。

| プラン名 | 月額料金 (USD) | 特徴 |
| --- | --- | --- |
| Basic | $5 | ファイル数・文字数制限緩和、広告非表示、基本的な機能 |
| Pro | $15 | 高度な機能、ストレージ拡張、優先サポート |
| Business | $50 | チーム機能、高度なセキュリティ設定 |

### 6.3. 広告モデルの導入

無料プランの収益性を補強し、APIコストを賄うため、バナー広告を導入します。

- **配置の限定**: 編集画面ではなく、ファイルリスト画面や設定画面の下部に限定。
- **ユーザーへの配慮**: 静的なバナー広告のみとし、執筆の邪魔をしない。
- **有料プランへの誘導**: 「広告非表示」を有料プランの主要なメリットの一つとして訴求。

## 7. 技術構成案と開発方針

### 7.1. 基本構成

- **フロントエンド**: React Native (iOS先行)
- **バックエンド**: AWS Lambda / Google Cloud Functions (APIプロキシ)
- **バージョン管理**: Git (GitHub API経由)
- **認証**: Firebase Authentication (GitHub OAuth, 将来的にGoogle/Apple/メール追加)
- **ローカルファースト**: データの一次保存はデバイス内。GitHubは同期・バックアップ用。

### 7.2. メンテナンス性と拡張性を高める開発方針

- **TypeScriptの全面採用**: 厳密な型定義により、大規模化してもコードが破綻しにくい設計にする。
- **クリーンアーキテクチャの意識**: ビジネスロジックとUI、外部API連携を分離し、将来的なプラットフォーム変更やAPI変更に強い構造にする。
- **自動テストの導入**: 重要なロジック（Markdown処理、同期ロジック等）にはユニットテストを記述し、安定性を確保する。
- **CI/CDの構築**: GitHub Actions等を利用し、コードの品質チェックとデプロイを自動化する。

## 8. UI/UX設計思想と画面イメージ

### 8.1. メイン画面（ファイルリスト画面）

- **同期ステータスアイコン**: ローカルとクラウドの同期状態を視覚的に表示。
- **広告エリア**: 無料プラン時のみ画面下部に表示。

### 8.2. 編集画面

- **自動保存機能**: 入力を止めると自動でローカルに保存。
- **Markdownツールバー**: キーボード上に配置し、記法を覚えていなくても入力可能。
- **集中モード**: 執筆中にUIを隠し、コンテンツに没入できる環境を提供。

### 8.3. 変更履歴（差分ビュー）画面

- **タイムライン形式**: 変更履歴を時系列で表示。
- **簡単ロールバック**: 過去のバージョンをプレビューし、ワンタップで復元可能。

## 9. 今後の展望

- **Android/Web展開**: React Nativeのコードベースを最大限活用。
- **認証方式の拡充**: GitHub以外のユーザーも利用可能にする。
- **AIエージェント**: ドキュメントの内容を理解し、自律的に提案を行う機能。
